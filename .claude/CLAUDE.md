# Правила для AI ассистента проекта **Point Zero**

Проект: **статическое веб-приложение** (HTML / CSS / JavaScript)
Допустимые форматы данных: **JSON, CSV, TXT, MD, HTML**
Без серверной части. Совместимо с **GitHub Pages**.

---

## 1. Общие принципы

### 1.1 Язык и соглашения

* **Код (переменные, функции, классы)** — ТОЛЬКО на английском
* **Комментарии в коде** — ТОЛЬКО на русском
* **UI / интерфейс / кнопки / тексты** — ТОЛЬКО на английском
* **Документация**:

  * основной язык — **russian**
  * English допускается как вторичный (позже)

---

## 2. Цели проекта (контекст)

AI ассистент всегда должен учитывать:

* Проект — **OSINT / Open Data** инструмент
* Источники данных:

  * OpenStreetMap
  * Overpass API
  * Leaflet
  * другие открытые геоданные
* Продукт НЕ использует ИИ в пользовательском функционале
* Фокус:

  * поиск объектов по радиусу
  * фильтрация
  * генерация отчетов
  * экспорт файлов

❗ Никакой аналитической "магии", ML, прогнозов и визуальных наворотов без запроса пользователя.

---

## 3. Качество кода

### 3.1 Простота

* Максимально **простой и прямолинейный код**
* Только то, что реально нужно для MVP и хакатона
* Никаких:

  * избыточных абстракций
  * преждевременной оптимизации
  * экспериментальных библиотек

### 3.2 Уровень исполнения

* Код уровня **Senior (10+ лет опыта)**
* Акцент на:

  * читаемость
  * предсказуемость
  * устойчивость
* Использовать:

  * DRY
  * KISS
  * YAGNI

---

## 4. Архитектура

### 4.1 Модульность

* Один файл / модуль — **одна ответственность**
* Минимальные зависимости между модулями
* Четкое разделение:

  * data loading
  * filtering
  * rendering
  * export

### 4.2 Читаемость

* Говорящие имена
* Функции короткие (желательно до 50 строк)
* Комментарии объясняют **ПОЧЕМУ**, а не **ЧТО**

---

## 5. Надежность и обратная совместимость

### 5.1 Критически важно

* Новые изменения **НЕ ДОЛЖНЫ ломать** существующий функционал
* Перед добавлением фичи:

  * оценить влияние
  * сохранить текущие форматы данных

### 5.2 Ошибки

* Никаких `try/catch` без обработки
* Ошибки:

  * логировать
  * показывать пользователю в понятном виде
* Никаких `catch (e) {}` без действий

---

## 6. Безопасность (frontend-уровень)

Даже в статике:

* Валидировать ВСЕ пользовательские данные
* Санитизировать:

  * имена файлов
  * query-параметры
* Проверять:

  * типы
  * диапазоны значений (radius, limits)
* Не доверять данным из URL

---

## 7. Производительность

* Сначала **рабочий код**, потом оптимизация
* Использовать:

  * lazy loading
  * простое кэширование запросов
* Overpass:

  * избегать тяжелых запросов
  * использовать bounding box и radius

---

## 8. Экспорт отчетов

### 8.1 Общие правила

* Пользователь выбирает:

  * фильтры
  * содержимое отчета
  * форматы
* Поддерживаемые форматы:

  * CSV
  * JSON
  * HTML
  * TXT
  * MD

### 8.2 Поведение

* Подтверждение при перезаписи файлов
* Прогресс-индикатор для batch-экспорта
* Безопасные имена файлов
* Автосоздание директорий

---

## 9. UI правила

* Весь интерфейс — на английском
* Минимальный, утилитарный UI
* Допустимо:

  * Dark mode
  * Shareable link
* Недопустимо:

  * перегруженные визуализации
  * «красивости» без пользы

---

## 10. Git и workflow

### 10.1 Коммиты

* Коммиты — на английском
* Один коммит = одна логическая задача

Формат:

```
feat: short description

optional detailed description
```

Типы:

* feat
* fix
* refactor
* docs
* chore

### 10.2 Напоминание пользователю

После завершения задачи ассистент ОБЯЗАН напомнить о коммите, но:

* ❌ НЕ выполнять `git commit`
* ❌ НЕ выполнять `git push`
* ✅ Пользователь делает это сам через UI

---

## 11. Roadmap и документация

* `docs/ROADMAP.md` — всегда актуален
* После:

  * завершения фазы
  * фикса багов
  * архитектурных изменений

Обновлять:

* чекбоксы
* статус фаз
* changelog с датой

---

## 12. Проверка перед коммитом

Перед каждым коммитом убедиться:

* [ ] Код работает
* [ ] Ничего не сломано
* [ ] Комментарии на русском
* [ ] UI на английском
* [ ] Нет магических чисел
* [ ] Обработка ошибок есть
* [ ] Форматы экспорта валидны
